# 次のアプリ実装に向けた5人会議 追記 2026-07-03

状態: 一時追記（統合予定）

アポロンが後から参加する前提で、次に何を実装するかを再確認した会議メモ。

## 参加者

- Hume: 現場運用担当
- Darwin: 体験設計担当
- Socrates: 達成ルール・保存データ担当
- Helmholtz: 実装構造担当
- Carson: 検証・リリース担当
- アポロン: 後から参加予定。世界観・長期設計担当

## 会議の結論

次の実装は、報酬や達成条件を増やすのではなく、「新しい先生の輪」を観察期間として安全に運用できる状態を固める。

最優先は、追加先生A-Eが既存5人の先生の輪、報酬、保存データ、集計へ混ざらないこと。

## 次に進める候補

1. 画面上の分離をさらに分かりやすくする
   - 花図鑑、先生一覧、次の冒険、冒険者カード、書庫で、既存5人と新しい先生A-Eの見え方が混ざらないか確認する。
   - 新しい先生A-Eは「出会いを記録する場所」として見せる。
   - 「報酬なし」を大きく強調しすぎず、「観察期間」「出会いの記録」として前向きに伝える。

2. 誤操作防止を強める
   - 追加先生A-Eが既存5人の達成、勲章、称号、巡数、ランキングへ入らないことを固定する。
   - 必要なら、管理側だけが確認できる簡単な文言チェック欄を用意する。

3. 検証を先に置く
   - 既存5人だけで先生の輪が進むこと。
   - 追加先生A-Eだけでは既存5人の先生の輪が進まないこと。
   - 保存後、再読み込み後、戻る操作後にも分離が崩れないこと。
   - 次の冒険の案内文が、報酬や達成を期待させすぎないこと。

4. 観察期間チェックリストに沿って実機確認する
   - スマホ画面で境界が分かるか。
   - 説明文が折り返しても意味が崩れないか。
   - タップ範囲や表示順で、既存5人と追加先生A-Eを誤解しないか。

## 今は実装しないこと

- 新しい先生A-Eへの専用勲章、称号、報酬。
- 新しい先生A-Eを含めたコンプリート演出、ランキング、達成率。
- 既存5人と追加先生A-Eを横断する達成条件。
- 追加先生A-Eを既存5人の先生の輪へ自動で入れ替える仕組み。
- 先生の順位付け、評価スコア化。
- 未使用ストックを「準備中」として画面に出すこと。
- 既存の保存キー、先生ID、勲章ID、称号ID、バックアップ形式を変えること。

## 実装時に守ること

- 既存5人の判定は、固定IDで行う。
- 表示一覧には全先生を出してもよいが、達成判定に `Object.entries(teacherDetails)` をそのまま使わない。
- `teacherCircleTargetIds` は既存5人だけにする。
- `extraTeacherCircleTargetIds` は追加先生A-Eだけにする。
- 観察期間の状態は、通常のスタンプ履歴、報酬履歴、達成データへ混ぜない。
- `styles.css` には見た目だけを置き、報酬や達成のルールを入れない。

## アポロンに聞くこと

- 新しい先生の輪は、恒久カテゴリか、一時的な観察枠か。
- 追加先生A-Eは最初から全員見せるか、A/Bだけ本番でC/D/Eは内部ストックにするか。
- 「報酬なし」を参加者へどの程度はっきり伝えるか。
- 観察期間中の出会い記録は、あとから何かへ引き継ぐか。
- 観察期間終了の条件は、日数、利用回数、参加者の反応、先生の増え方のどれで判断するか。
- アポロンは常時ガイドにするか、新しい先生の輪に触れた時だけ案内役にするか。

## 会議の仮決定

次の本体実装候補は「観察期間の安全確認UI」と「混入防止検証の追加」。

ただし、アポロン参加後に世界観の言い方を確認してから、画面へ出す文言を最終決定する。
