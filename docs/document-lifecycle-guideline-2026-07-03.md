# 文書ライフサイクル指針 2026-07-03

状態: 正式保存

docs の文書が増えてきたときに、各文書の成熟度と扱いを迷わないための指針。

## 目的

議事録、追記メモ、設計指針、作業ログが増えても、後から読んだ人が次のことをすぐ判断できるようにする。

- これは正式な判断か。
- これは後で統合する一時メモか。
- これはまだ会議前のアイデアか。

## 状態の種類

文書の冒頭に、できるだけ次のいずれかを書く。

### 正式保存

今後の基準、またはその時点で採用された設計判断として残す文書。

例:

```text
状態: 正式保存
```

### 一時追記（統合予定）

本体の議事録、設計指針、作業ログなどへ後で統合する予定の文書。

例:

```text
状態: 一時追記（統合予定）
```

### 検討中

会議前のアイデア、草案、まだ採用されていない案。

例:

```text
状態: 検討中
```

## 運用メモ

- 状態行は文書の冒頭、タイトルのすぐ下に置く。
- 絵文字は使ってもよいが、状態名そのものは統一する。
- 状態が変わったら、理由も短く残す。
- 一時追記のまま長期間残っている文書は、Archivist が統合先を確認する。
- 正式保存にする前に、「これは将来の基準として読まれてよいか」を確認する。

## Archivist: 記録・継承担当

プロジェクトの判断・理由・歴史が将来も理解できるように整理し、適切な文書へ保存する。

コードではなく、判断の継承を担当する。

## 三者の役割

このプロジェクトでは、次の役割分担が自然に育ってきた。

- 利用者と現場の代表: 実際に水曜会で使われる体験を知り、現場の違和感や希望を伝える。
- 設計者・実装者: 判断を安全に形へし、既存データと世界観を守りながら実装する。
- 長期設計・Archivist: 世界観や判断の積み重ねを見守り、将来の自分たちが迷わないように記録する。

この役割分担は固定された肩書きではなく、開発を続ける中で育った協力の形である。

## 今日の節目

今日、コードだけでなく、記録そのものにもライフサイクルを持たせた。

これは小さな書き方の変更ではなく、プロジェクトの歴史を追えるようにするための設計判断である。

この文書は、未来の自分たちが迷わないための約束として扱う。

## コード・記録・記憶の三層

このプロジェクトは、次の三つの層で育てる。

### コード

アプリが何をするかを決める層。

例:

- `index.html`
- `script.js`
- `styles.css`

### 記録

何を決めたかを残す層。

例:

- 議事録
- 作業ログ
- 設計判断

### 記憶

なぜそれを大切にしたかを受け継ぐ層。

例:

- `project-guiding-principles.md`
- `document-lifecycle-guideline-2026-07-03.md`
- Archivist の役割

コードだけでは、なぜその仕様を守るのかまでは分からない。

記録だけでは、その考えが今も有効かまでは分からない。

記憶だけでは、実際にどう実装されたかまでは分からない。

三つがそろって初めて、プロジェクト全体の知識になる。
