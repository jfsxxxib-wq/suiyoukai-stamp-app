# 追加先生 実装前設計メモ 2026-07-02

目的: 先生が2人増えた時に、既存5人の記録・先生の輪・バックアップを壊さずに追加できるよう、実装前の方針を固定する。

関連:

- `docs/future-teacher-stock-2026-07-02.md`
- `docs/future-teacher-stock-preview-2026-07-02.png`

## 結論

- 既存5人は固定する。
- 新先生2人は、まず「追加先生枠」として扱う。
- 既存の先生の輪は、当面「基本5人の輪」のまま維持する。
- 新先生は、先生の輪にすぐ混ぜない。
- 旧バックアップに新先生IDがない時は、0回として補完する。
- 花と妖精は、1先生につき3巡分をセットで割り当てる。
- 追加先生用ストックは、利用者画面にはまだ出さない。

## 追加先生の仮枠

| 仮枠 | 並び | 扱い | 花・妖精 |
| --- | --- | --- | --- |
| 新先生A | 既存5人の後ろ | 追加先生枠 | すずらん、木蓮、ひなげし |
| 新先生B | 新先生Aの後ろ | 追加先生枠 | 白詰草、蘭、花水木 |

実際の先生名・段位・写真が決まるまでは、仮IDでアプリ本体に入れない。

## 並び方

初期実装:

1. 既存5人を現在の順で固定する。
2. 新先生Aを6番目に置く。
3. 新先生Bを7番目に置く。

将来検討:

- 「今日来ている先生」を上に出す。
- 「通常の先生」と「追加先生」を見出しで分ける。
- 先生が8人以上になったら、絞り込みを検討する。

今は五十音順にはしない。既存ユーザーの見慣れた順を守る。

## 花・妖精割り当て

| 仮枠 | 5回 | 10回 | 15回 |
| --- | --- | --- | --- |
| 新先生A | すずらん × ネズミ妖精 | 木蓮 × 白猫妖精 | ひなげし × リス妖精 |
| 新先生B | 白詰草 × トイプードル妖精 | 蘭 × 亀妖精 | 花水木 × 鶴妖精 |

注意:

- 花水木の鶴は、別の妖精羽を付けず、鶴自身の翼を生かす。
- 白詰草の仲間は、狐ではなくトイプードルにする。
- 勲章は先生個別では増やさない。
- 称号は、割り当て時に「先生名 + 巡目 + 花名」で作れる形にする。

## データ方針

### 先生ID

既存先生IDは変更しない。

既存:

- `tsuneishi`
- `yuki`
- `koike`
- `yamashiro`
- `matsumoto`

追加時の仮ID例:

- `teacher_extra_01`
- `teacher_extra_02`

実先生名が決まったら、英数字の安定IDを決める。あとで変更しない。

### 保存データ

`userProgress.stamps.teacherLessonCounts` に、新先生IDを追加する。

旧バックアップや旧端末データでは、新先生IDが存在しない可能性がある。その場合は復元・読み込み時に0回で補完する。

例:

```json
{
  "teacherLessonCounts": {
    "tsuneishi": 0,
    "yuki": 0,
    "koike": 0,
    "yamashiro": 0,
    "matsumoto": 0,
    "teacher_extra_01": 0,
    "teacher_extra_02": 0
  }
}
```

### 先生の輪

現在の `teacherCircleRounds` は、基本5人だけで計算する。

新先生を混ぜると、既存ユーザーの「先生の輪」が後退して見える可能性があるため、追加先生は別グループで扱う。

将来案:

- `teacherCircleGroups.primary`: 既存5人
- `teacherCircleGroups.extra`: 新先生枠

最初の実装では、保存形式にグループを増やす前に、まず設計と検証だけ作る。

## バックアップ方針

現状のバックアップ形式:

- `appId`: `suiyoukai-stamp-adventure`
- `formatVersion`: `1`
- `progress.schemaVersion`: `2`

追加先生実装で必要なこと:

- 旧バックアップに新先生IDがなくても復元できる。
- 復元時に新先生IDを0回で補完する。
- 先生の輪は基本5人で再計算する。
- 追加先生の0回が混ざって、既存の先生の輪を下げない。

形式変更を入れる場合:

- `formatVersion` を上げる。
- 旧形式から新形式への移行ルールを明記する。
- 旧バックアップ復元テストを作る。

## 実装順

### 1. 設計だけ

- このメモを保存する。
- ストック表を保存する。
- 先生追加時の検証項目を決める。

### 2. 検証を先に作る

アプリ本体に先生を入れる前に、次の検証を作る。

- 旧バックアップに新先生IDがなくても0回で補完できる。
- 既存5人の先生の輪が後退しない。
- 既存5人の花図鑑・冒険者カード表示が変わらない。
- 新先生枠を追加しても、横幅やスマホ表示が壊れない。

2026-07-02時点の確認:

- `docs/verify-additional-teachers-readiness-2026-07-02.cjs` を追加した。
- 結果は `docs/additional-teachers-readiness-2026-07-02/result.json`。
- 基本5人の先生ID、先生の輪の計算対象、バックアップ形式、通常読み込み時の補完、先生の輪の必要人数は確認済み。
- 旧バックアップ復元は、先生追加前に補完ルールの実装が必要。

### 3. データ補完を入れる

- 読み込み時に新先生IDがなければ0回で追加する。
- バックアップ復元時も同じ補完を通す。
- 管理画面の下書きにも新先生IDを補完する。

### 4. 表示を入れる

- 先生一覧に新先生を追加する。
- 先生詳細に花・妖精3巡分を持たせる。
- 花図鑑に出すのは、状態が `active` になってからにする。

### 5. 実地確認

- スマホ幅430pxで先生一覧が長くなりすぎないか見る。
- 先生詳細へ入れるか見る。
- 既存先生の押印と報酬が変わらないか見る。
- 旧バックアップ復元を実行する。

## まだ実装しないこと

- 新先生を基本5人の先生の輪へ混ぜる。
- 準備中の花を利用者画面に出す。
- 新しい勲章を大量に増やす。
- 先生一覧の五十音順化。
- 花・妖精の個別本番素材化。
