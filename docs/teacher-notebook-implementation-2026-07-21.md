# 「一局のご縁帳」実装メモ（2026-07-21）

## Ver.1 完成・保管

2026年7月21日、先生ページ Ver.1 は現在の方向で一度完成とする。

以後は新機能を急いで増やさず、Ver.1の温かさと静かな見え方を保つ。次段階では、先生本人だけが見られる安全な認証、参加者本人の同意、対局記録と会場情報の安全な連携を中心に検討する。

公開、GitHubへの保存、参加者向けアプリへの接続は、別途明確な確認を得るまで行わない。

## 今回追加したもの

- 参加者向けアプリとは別入口の `teacher-notebook.html`
- 表紙、今日の対局、参加者との対局履歴、再会表示の4画面
- 同意済み記録だけに氏名・対局日・棋力・置き石を表示する整形処理
- 同意なしの記録を「お名前は非表示」とし、人数にだけ含める処理
- 認証済み先生IDと一致する記録だけを残すクライアント側の防御
- 認証情報がない場合、記録を一切表示しない閉鎖画面
- 検索エンジンに載せない `noindex / noarchive`

参加者向けの `index.html`、`script.js`、`styles.css` には手を加えていない。

## 確認用入口

見本データでの確認用。公開用URLとしては使用しない。

- 表紙: `teacher-notebook.html?preview=1&teacher=tsuneishi`
- 再会: `teacher-notebook.html?preview=1&teacher=tsuneishi&participant=p-yamada`
- 別の先生の分離確認: `teacher-notebook.html?preview=1&teacher=yuki`
- 閉鎖画面: `teacher-notebook.html`

見本モードには常に「見本データ」と表示する。

## 本番接続で必須のもの

現在の対局記録は参加者本人の端末内 `localStorage` にあり、別端末の先生ページへ自動集約する仕組みはない。したがって、今回のページは本番データへ未接続である。

本番では、ログイン完了後にサーバーから次の形のセッションを渡す。

```js
window.SUIYOUKAI_TEACHER_SESSION = {
  teacher: { id: "認証済み先生ID", name: "表示名" },
  getRecords: async ({ teacherId }) => {
    // 認証済み本人の記録だけをサーバーから取得する
  },
};
```

重要: ブラウザ側の `teacherId` 絞り込みだけを閲覧権限として扱わない。サーバー側でも、ログイン中の先生IDと各対局の先生IDが一致する場合だけ返す。

## 本番データの最小項目

- `teacherId`
- 参加者を同じ先生との再会だけに結び付ける内部ID
- 参加者の表示同意（真偽値）
- 同意時のみ表示する参加者名
- 対局日
- 棋力
- 置き石
- 会場名

勝敗と感想は、必要性を改めて判断するまで先生の日常画面と先生ページ用データから外す。

### 対局場所の扱い

対局場所は、再会を助ける記録として将来表示したい方向とする。

- 各対局ごとに、その時の会場名を保存する
- 「前回の場所」は最新記録ではなく、前回の対局記録に保存された会場名を表示する
- 今回と前回で場所が違う場合も、前回の実際の会場名が分かるようにする
- 場所の記録がない場合は、場所に関する文章を出さず「今回で2回目です」だけを表示する
- 会場名は自由入力ではなく、運営が登録した会場一覧から選択することを基本とする。必要な場合のみ新規会場を追加する

現在の参加者向けアプリとの連携方法が決まるまでは、日常画面への本実装は保留する。

住所、電話番号、メールアドレスは先生ページ用データへ入れない。

## 参加者側との連携方針

対局QRの反映時、参加者本人に「対局した先生に名前と対局記録を表示してよい」の同意状態が必要になる。同意した記録だけを先生IDにひも付けてサーバーへ送る。同意しない場合は、氏名・棋力・感想などを送らず、匿名の対局1件として扱う。

先生自身の入力を通常フローに入れない。受付記録または参加者スマホでの先生QR反映を起点にする。

## 権限の境界

- 先生: 認証済みの自分の `teacherId` の記録のみ
- 参加者: 自分の端末・自分の記録のみ
- 運営: 運営に必要な管理権限。先生ページと入口を分ける
- 一般: 公開用プロフィール以外は不可

先生ページ内には、ほかの先生へのリンク、先生一覧、ランキング、比較、人数比較を置かない。
