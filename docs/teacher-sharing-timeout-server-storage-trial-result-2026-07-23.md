# 先生ページ タイムアウト時サーバー保存実機試験結果 2026-07-23

## 結論

試験専用Apps Scriptと公開済みの独立試験ページだけを使い、15秒タイムアウト時のサーバー側保存状況を実機確認した。

判定は合格とする。

- ブラウザーは15秒でタイムアウト表示になった
- 初回送信後に自動再送は起きなかった
- Apps Script側では、ブラウザーがタイムアウトする前に対象IDが1件保存されていた
- 同じ`requestId`・`recordId`を手動再送しても、保存は1件のままで重複しなかった
- ブラウザーがタイムアウトした時点では、画面だけから保存済み・未保存を断定できないことを実機で確認した

## 今回わかった重要点

ブラウザーのタイムアウトは「未保存」を意味しない。

今回の実機試験では、ブラウザーは応答を受け取れずタイムアウトしたが、Apps Script側では保存済みだった。したがって、タイムアウト画面で「未保存」と断定すると誤案内になる。

## 画面文言の候補

通常アプリにはまだ反映しない。事前確認用の候補として、次を提案する。

### 候補A

見出し:

`送信結果を確認できませんでした`

本文:

`15秒以内に応答を受け取れませんでした。サーバー側では保存済みの可能性があります。自動再送はしていません。運営者に確認してください。`

### 候補B

見出し:

`通信がタイムアウトしました`

本文:

`保存されたかどうかを、この画面では確認できませんでした。重複を防ぐため自動再送はしていません。`

今回の事実を最も誤解なく伝えるのは候補A。ただし、通常アプリへの反映は悦子さんの明確な承認後に行う。

## 最終判定

合格。

タイムアウト時にはブラウザー側で成功・未保存のどちらも断定せず、サーバー照合が必要である。サーバー側の`recordId`重複防止は、同じ`requestId`を手動再送した場合にも有効だった。
