# Project Guiding Principles

## Protected and flexible areas

Some parts of the project should be protected because they define the character and continuity of the app.

Protected areas:

- The world view that helps Water Wednesday memories grow
- The basic five-teacher circle
- Compatibility with existing user data and backups
- The habit of recording why decisions were made
- Improvements based on actual user behavior

Flexible areas:

- UI design
- Visual effects
- Screen order and layout
- How new teachers are added
- New flowers, fairies, medals, and events
- Operation tools, as long as they protect existing records

When changing a flexible area, check that it does not damage a protected area.

この文書は実装手順ではありません。議事録より上位に置く、プロジェクトの考え方をまとめる場所です。機能や画面は変わっても、判断の基準はここに残します。

## 大切にすること

このアプリは、単に便利な道具を目指すだけではありません。

開いたときに、その日の先生、対局、花、妖精、参加した時間を思い出せる場所にします。

新しい機能を考えるときは、最初に次の問いを確認します。

> この機能は、人の思い出を育てるだろうか？

便利でも、水曜会の記憶を育てないものは優先しません。

## 会議・設計判断で残す項目

会議メモや設計判断メモには、できるだけ次の項目を残します。

- 決定事項
- 決めた理由
- 今は作らないもの
- 後で見直す条件
- 世界観との整合
- 利用者から学んだこと

決定事項だけでは、数か月後に判断を再現できません。なぜその道を選んだか、なぜ別の道を選ばなかったかを残します。

## 何を追加しないかも記録する

作らないことも設計判断です。

あとで「なぜ作らなかったのか」が分からなくならないように、見送った機能も理由と一緒に残します。

記録する内容:

- 見送った案
- 見送った理由
- いつ、どんな条件なら見直すか
- 利用者の体験にどう影響するか

## 利用者から学ぶ

実際の利用者の反応は、設計判断の重要な材料です。

例:

- 背景のマップマークを何度も押していた
- スマホでは登録方法が分かりづらかった
- 妖精が小さくて気付きにくかった
- 先生紹介が下にあると後回しに見えた

こうした反応は、思いつきの変更ではなく、利用者から教わった改善として記録します。

## 世界観との整合

このアプリでは、画面や機能が水曜会の雰囲気から外れないことを重視します。

判断に迷ったときは、次を確認します。

- 花、妖精、先生、対局の記憶につながるか
- 使う人にとって自然な順番か
- 説明されなくても意味が伝わるか
- 便利さだけが前に出すぎていないか
- 水曜会らしい温度が残っているか

## 議事録との役割分担

議事録は、その日の検討と判断を残します。

この文書は、複数の議事録をまたいで残す判断基準です。

議事録に新しい判断軸が出てきた場合は、必要に応じてこの文書へ昇格させます。

## 引き継ぎのために

コードはアプリの動きを残します。

設計判断メモは、なぜその形にしたかを残します。

将来このプロジェクトを引き継ぐ人が、水曜会らしさを再現できるように、判断の理由と利用者から学んだことを残します。
