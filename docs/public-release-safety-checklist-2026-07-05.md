# GitHub Pages公開前 安全チェックリスト 2026-07-05

状態: 準備中

目的: GitHub Pagesで公開しても、参加者向けアプリとして大きな危険がないか確認する。

## 最小安全対応

- 公開用コードから固定の運営用パスコードを削除した。
- 運営用パスコードは、運営端末の初回起動時にその端末へ設定する形にした。
- 公開URLに `#admin` を付けても、管理画面が直接開かないようにした。
- 管理画面は、通常画面から簡単には見えない入口にした。
- 強いオフラインキャッシュはまだ入れない。

## 公開前に確認すること

- コード内に実際の運営用パスコードが残っていない。
- 参加者が通常URLを開いた時、管理タブが見えない。
- `公開URL/#admin` を開いても管理画面が直接開かない。
- 運営端末では、隠し入口から管理画面を開ける。
- 運営端末の初回だけ、8文字以上のパスコードを設定できる。
- 2回目以降は、設定したパスコードで解除できる。
- 受付QR、先生ごとの固定QR、対局内容QRは、運営端末で作成する。
- 本番URLで作ったQRを、参加者スマホで読む。

## まだ完全なセキュリティではないこと

GitHub Pagesは静的な公開サイトなので、サーバー側の本格的な認証はない。

そのため、今回の対応は次の位置づけにする。

- 参加者が普通に使う範囲で、管理画面へ簡単に触れないようにする。
- コード内に本物の運営パスコードを置かない。
- 8/30に間に合う最小安全対応にする。

本格的に運営データを守る段階では、Supabaseなどの外部保存と認証を検討する。

## 公開してよい判定

次を満たせば、8/30向けの最小公開として進めてよい。

- 固定パスコードがコード内にない。
- 通常URLから管理画面が見えない。
- `#admin` で直接管理画面へ行けない。
- 運営端末では管理画面を開ける。
- 本番URLでホーム画面追加ができる。
- 本番URLでQR運用を再確認できる。
