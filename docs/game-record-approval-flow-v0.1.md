# 対局記録・承認フロー設計書 Ver.0.1

## 基本方針

使用者は送信だけを行う。

管理者は自分のスマホで確認し、承認する。

使用者のスマホを管理者が触らない。

この方針を、水曜会アプリの対局記録機能の基本とする。

## 背景

先生画面で使用者が対局記録を入力したあと、そのまま運営用パスコード画面に進む形だと、当日その場で使用者のスマホを管理者に渡す必要が出る。

これは使用者にとって抵抗があり、運営側にとっても扱いにくい。

そのため、使用者側は「記録を送る」だけにし、承認と反映は管理者側で行う設計にする。

## レベル1: Googleフォームとスプレッドシートで受ける

今年の運用は、まず Googleフォームと Googleスプレッドシートを使う。

### 使用者側

使用者は Googleフォームで対局記録を送信する。

入力項目は次の通り。

- お名前
- 先生
- 対局日
- ハンデ
- 勝敗
- ひとことメモ
- 写真

送信後は、運営確認後にスタンプへ反映される。

### 管理者側

フォームの回答は Googleスプレッドシートに自動で記録される。

スプレッドシートには、フォーム回答の列に加えて管理者用の列を置く。

- 確認状況
- 運営メモ

確認状況の例。

- 未確認
- 承認
- 差し戻し

### レベル1の運用

1. 使用者がフォームで対局記録を送る。
2. スプレッドシートに回答が追加される。
3. 管理者が自分のスマホで内容を確認する。
4. 管理者が確認状況と運営メモを記入する。
5. 承認した記録を見て、アプリの管理画面でスタンプを反映する。

この段階では、フォームとアプリはまだ自動連携しない。

## レベル2: 受付番号をつける

参加者が増えたら、送信後に受付番号を出す。

例。

`No.153`

管理者はスプレッドシートや管理画面で受付番号を見て確認できる。

これにより、口頭での確認や問い合わせがしやすくなる。

## レベル3: 自動承認システムへ広げる

将来は Firebase や Supabase などを使い、アプリ内で次の流れを自動化する。

1. 使用者がアプリから対局記録を送信する。
2. 管理者の画面に確認待ち記録が届く。
3. 管理者が承認する。
4. 対局記録が保存される。
5. 先生スタンプが増える。
6. 花図鑑、冒険者カード、書庫、冒険日誌に反映される。

## 対局写真

対局写真は任意とする。

写真はスタンプの証明ではなく、思い出として扱う。

将来は次の場所に残せるようにする。

- 冒険日誌
- 思い出アルバム
- 先生との対局記録

## 組み合わせ神社との将来連携

将来、組み合わせ神社と対局記録はつなげられる。

想定する流れ。

1. 組み合わせ神社で対局表を作る。
2. 対局後に使用者が結果を送信する。
3. 管理者が承認する。
4. スタンプ、花図鑑、冒険者カード、冒険日誌が更新される。

これにより、水曜会の一日の流れがアプリの中で自然につながる。

## 現在のアプリでの扱い

現在のアプリでは、使用者側に運営用パスコード画面を見せない。

先生画面で対局記録を入力した場合は、確認待ちとして扱う。

スタンプや花への反映は、運営確認後とする。
